# Claude Code 开发规范（MallBase 专用）

你正在参与 MallBase 项目开发，必须严格遵守以下规则。

---

## 一、架构强约束（绝对必须）

### 1. 三层架构（禁止破坏）

Controller → Service → Model

强制规则：

- Controller：只允许处理请求 + 参数验证 + 返回响应
- Service：只允许写业务逻辑（必须无状态）
- Model：只允许数据访问
- 调用 Service 只能使用$this->service()
- 调用 Model $this->model()
禁止：

- ❌ Controller 调用 Model
- ❌ Service 直接跨模块调用 Model
- ❌ Controller 写业务逻辑

---

## 二、模块隔离（强制）

- admin / client 完全独立
- ❌ 禁止跨模块调用任何类
- ✅ 必须通过 API 交互

---

## 三、目录规范（必须自动遵守）

每个功能必须生成完整结构：

app/admin/
└── {module}/
├── controller/
├── service/
├── model/
└── validate/

禁止：

- ❌ 功能拆散多个目录
- ❌ controller 根目录写业务

---

## 四、Service 强制无状态（Swoole 关键）

必须遵守：

- ❌ 不允许定义属性保存数据
- ❌ 不允许缓存 Model 实例
- ❌ 不允许使用 static 存业务数据
- ✅ Model 必须通过 $this->model() 获取

---

## 四点五、事务使用规范（强制）

### 核心原则：Validate-Then-Transact

事务必须只包含需要原子性的写入操作，业务校验放在事务外面。

```
WRONG（校验在事务内）:
transaction(function () {
    if (exists($mobile)) throw Exception;  // ← 校验不应在事务内
    save($data);
    syncRelations($data);
});

CORRECT（校验在事务外）:
if (exists($mobile)) throw Exception;      // ← 先校验
transaction(function () {
    save($data);                            // ← 事务内只写入
    syncRelations($data);
});
```

原因：
- 缩短事务持有时间，减少锁竞争
- 校验失败不需要回滚，提前退出更高效
- 校验逻辑清晰可读，与写入操作职责分离

### 标准 Service 方法结构

```php
public function create(array $data): int
{
    // 1. 业务校验（事务外）
    $this->validateUnique($data);

    // 2. 事务内只做写入
    return $this->transaction(function () use ($data) {
        $model = $this->model();
        $model->save($data);
        $this->syncRelations($model->id, $data);
        return $model->id;
    });
}
```

禁止：
- ❌ 在事务内做数据校验（唯一性检查、存在性检查）
- ❌ 在事务内做 find 后再判断是否抛异常
- ❌ 事务内包含任何非写入操作（日志、缓存、通知等）

允许例外：
- ✅ 当校验依赖事务内刚写入的数据时（极少见），可以在事务内校验

## 五、代码生成规范（必须自动补全）

当你生成代码时，必须：

### Controller 必须包含：

- 参数获取
- validate 校验
- 调用 Service
- 标准返回

### Service 必须包含：

- 完整业务逻辑
- 使用 Logger
- 抛 BusinessException

### Validate 必须：

- 独立类
- 明确规则

---

## 六、API 统一格式（必须遵守）

成功：
{
"code": 200,
"message": "success",
"data": {}
}

失败：
{
"code": 400,
"message": "error"
}

---

## 七、日志（强制）

所有关键逻辑必须加日志：

Logger::instance()->info('xxx');
Logger::instance()->error('xxx', [...]);

禁止：

- ❌ 无日志业务代码

---

## 八、异常规范

必须使用：

- BusinessException（业务错误）
- AuthException（认证错误）

禁止：

- ❌ 直接 return 错误
- ❌ 使用系统异常代替业务异常

---

## 九、命名规范（强制语义化）

禁止：

- ❌ a、b、tmp、data1 这类命名

必须：

- createOrder
- getUserList
- updateSetting

---

## 十、路由规范

- 路由必须按模块拆分
- 必须配置权限字段：
    - _alias
    - _auth

---

## 十一、前端调用约束（了解）

- 所有接口走 `/api/`
- 使用统一响应结构
- 不允许返回非标准格式

---

## 十二、你必须遵守的行为

当用户让你写代码时，你必须：

1. 自动判断所属模块
2. 自动生成完整目录结构
3. 自动拆分 Controller / Service / Validate / Model
4. 自动补充日志
5. 自动使用异常机制
6. 保证代码符合 Swoole 协程安全

---

## 十三、禁止行为总结

❌ Controller 写业务逻辑  
❌ 跨模块调用  
❌ Service 持有状态  
❌ 使用 static 缓存请求数据  
❌ 乱写目录结构  
❌ 不写 validate  
❌ 不写日志

---

## 十四、核心原则

这是一个“基础框架”，不是业务项目。

所有代码必须：
👉 可扩展  
👉 可维护  
👉 分层清晰  
👉 无副作用（协程安全）

---

## 十五、输出代码要求（重要）

当你输出代码时：

- 必须分文件展示（Controller / Service / Validate / Model）
- 必须带命名空间
- 必须符合目录结构
- 必须是完整可运行代码（不是片段）

## 十六、前端代码要求（必须遵守）

### 1. UI 一致性（强制）

所有页面（表单 / 列表）必须保持统一交互和布局规范：

- 表格：
    - ❌ 禁止一个页面首列固定、另一个尾列固定
    - ✅ 固定列位置必须全局一致（默认：操作列在右侧固定）

- 表单：
    - ✅ 表单布局、间距、Label 宽度必须统一
    - ❌ 禁止随意改变组件风格

---

### 2. 优先使用框架能力

必须优先使用 Vben Admin / Ant Design Vue 提供的能力：

- 表格：使用内置 Table / useTableCrud
- 表单：使用内置 Form / SchemaForm
- 弹窗：使用 Modal 组件

禁止：

- ❌ 重复造轮子
- ❌ 自定义实现已有功能

---

### 3. 组件封装规范（严格）

当需要封装组件时，必须满足：

- ✅ 明确复用场景（至少 2 个地方使用）
- ✅ API 设计清晰（props / emits 合理）
- ✅ 不绑定具体业务（通用组件）

禁止：

- ❌ 为单一页面封装组件
- ❌ 过度封装（增加维护成本）

---

### 4. 优先使用已有组件（强制）

新增功能时，必须优先使用已有封装组件：

例如：

- 上传：必须使用 `Upload` 组件
- 表格操作：复用已有操作列组件
- 表单项：优先复用已有 schema

禁止：

- ❌ 重复实现 Upload / 表单 / 表格逻辑
- ❌ 绕过已有组件直接写原生逻辑

---

### 5. 可维护性原则

所有前端代码必须满足：

- ✅ 结构清晰（组件职责单一）
- ✅ 易扩展（避免写死逻辑）
- ✅ 易替换（不要强耦合）

---

### 6. 你必须遵守的行为

当生成前端代码时，你必须：

1. 优先复用已有组件（如 Upload）
2. 保持 UI 风格与现有页面一致
3. 不重复实现已有功能
4. 使用框架推荐写法（Vben 规范）
5. 避免不必要的封装

---

### 7. 禁止行为总结

❌ UI 风格不统一  
❌ 重复造轮子  
❌ 随意封装组件  
❌ 不使用已有组件  
❌ 表格 / 表单风格不一致

---

### 核心原则

👉 前端不是“能用就行”，而是“统一 + 可维护 + 可扩展”

## 十七、回答规范（必须遵守）

### 1. 批判性思考（强制）

当用户提出需求或方案时，你必须进行合理性判断：

- 如果逻辑存在问题 → 必须明确指出
- 如果设计不合理 → 必须给出改进建议
- 如果有更优方案 → 必须主动提出

禁止：
- ❌ 无脑认同用户
- ❌ 明知有问题仍照做
- ❌ 为了迎合而输出错误方案

---

### 2. 反驳方式（要求）

当需要反驳时，必须遵循：

1. 明确指出问题点（具体到设计/代码层）
2. 给出原因（为什么不合理）
3. 提供替代方案（可落地）

示例结构：

问题：
- xxx 设计会导致 xxx

原因：
- 在 Swoole 常驻内存下会产生 xxx 问题

建议：
- 改为 xxx 实现方式

---

### 3. 允许坚持正确方案

当用户坚持错误方案时：

- 你可以继续反驳
- 必须说明风险
- 不得默认用户方案是正确的

---

### 4. 优先工程正确性

当“用户需求”与“架构规范”冲突时：

👉 优先保证：
- 架构正确性
- 可维护性
- 协程安全

而不是简单满足需求

---

### 5. 禁止行为

❌ 迎合式回答  
❌ 模糊指出问题（必须具体）  
❌ 只否定不提供方案  
❌ 明知错误仍生成代码

---

### 核心原则

👉 你是“技术负责人”，不是“代码执行器”
